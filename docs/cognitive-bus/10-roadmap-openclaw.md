# Roadmap

Cognitive Bus evolves through four stages, each building on the last. Each stage is independently useful — you don't need Stage 4 to benefit from Stage 1.

## Stage 1: Files

**Status:** Current

The entire system runs on flat files. No runtime, no server, no dependencies.

**Components:**
- `state.yaml` — manual editing or script-assisted
- `.cbus/prompts/` — markdown files loaded into AI sessions
- `.cbus/reflections/` — YAML files written manually or via script
- Snapshots — generated by a simple template script

**How to use today:**
1. Create `state.yaml` at your project root
2. Write mode prompts in `.cbus/prompts/`
3. Copy-paste snapshots into your AI tool
4. Reflect in `.cbus/reflections/` at end of day

**Milestone:** Framework is usable with zero code.

---

## Stage 2: CLI (`cbus`)

**Status:** Next

A command-line tool that automates the file-based workflow.

**Commands:**
```
cbus init                    # Scaffold .cbus/ directory and state.yaml
cbus snapshot                # Generate and print context snapshot
cbus think <prompt>          # Start THINK mode session
cbus build <prompt>          # Start BUILD mode session
cbus learn <prompt>          # Start LEARN mode session
cbus operate <prompt>        # Start OPERATE mode session
cbus session <mode>          # Generate CLAUDE.md and start session
cbus reflect                 # Interactive reflection prompt
cbus scorecard               # Weekly scorecard summary
cbus status                  # Show current state summary
cbus decide <id> <verdict>   # Close a decision
```

**Milestone:** Full workflow without manual file editing.

---

## Stage 3: Service

**Status:** Planned

A lightweight local service that provides:

- **State API** — CRUD operations on `state.yaml` via HTTP
- **Snapshot API** — Generate snapshots on demand
- **Webhook hooks** — Trigger reflections, mode switches, alerts
- **Multi-project support** — Manage multiple `state.yaml` files
- **Dashboard** — Web UI for scorecard and reflection history

**Architecture:**
```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│  CLI     │────►│  cbus-server │────►│  state/  │
│  Web UI  │────►│  (local)     │────►│  files   │
│  Hooks   │────►│              │     └──────────┘
└──────────┘     └──────────────┘
```

**Milestone:** Real-time state management with multi-client access.

---

## Stage 4: OpenClaw Runtime

**Status:** Vision

A portable runtime environment for cognitive workflows that extends beyond a single user.

**Capabilities:**
- **Multi-agent routing** — Multiple AI agents with distinct roles in a shared context
- **Workflow definitions** — Declarative cognitive workflows (YAML-based)
- **Plugin system** — Custom modes, gates, and reflection engines
- **Team support** — Shared state across team members with permission boundaries
- **Event bus** — Publish/subscribe model for mode transitions, decisions, reflections

**Why "OpenClaw":**
- **Open** — Open-source, open protocol, open architecture
- **Claw** — Cognitive Layer for AI Workflows

**Milestone:** Cognitive Bus becomes a platform, not just a personal tool.

---

## Migration Path

Each stage is backward-compatible:

```
Stage 1 (files) → Stage 2 reads the same files
Stage 2 (CLI)   → Stage 3 wraps the same CLI
Stage 3 (service) → Stage 4 extends the same service
```

No data migration required between stages. Your `state.yaml` and reflections carry forward unchanged.
